# Karma Build Tool

`kbuild` is the standardized build and project orchestration tool for
ktools-style workspaces and components. It is intentionally strict about
configuration, directory layout, and command combinations so build automation
stays predictable. It also enforces backend-owned hygiene checks for generated
artifacts that appear outside the intended `build/` trees, including
source-tree CMake configure/build output and Python bytecode/cache output in
Python source trees.

The shared tree supports multiple backend styles behind one command model.
Currently supported backend sections are:

- `cmake`
- `cargo`
- `java`
- `swift`
- `kotlin`
- `csharp`
- `javascript`

It is designed to be used as one shared implementation root, typically exposed
on your `PATH` as `kbuild` via a symlink such as:

```bash
ln -s /path/to/kbuild/kbuild.py ~/.local/bin/kbuild
```

## Quick Start

Bootstrapping a new project root from an empty directory:

```bash
kbuild --kbuild-config
# edit ./.kbuild.json
kbuild --kbuild-init
```

Normal day-to-day build commands:

```bash
kbuild --build-latest
kbuild --build-demos
kbuild --clean-latest
```

If the current project uses local `vcpkg`, prepare it first:

```bash
kbuild --vcpkg-install
```

## What Kbuild Manages

- strict validation of the effective kbuild config
- build slots under `build/<slot>/`
- SDK installation under `build/<slot>/sdk`
- ordered demo builds under `demo/*/build/<slot>/`
- backend-specific residual checks before build and git-sync operations
- project scaffolding from templates
- optional project-local `vcpkg`
- git initialization and simple sync flows
- backend dispatch for multiple language/build-system implementations

## Repository Layout

- `kbuild.py`: underlying entry script, typically exposed on `PATH` as `kbuild`
- `docs/`: detailed command, config, and workflow documentation
- `templates/`: repo scaffold templates generated by `--kbuild-init`

`kbuild --kbuild-init` currently scaffolds CMake-based projects only.
